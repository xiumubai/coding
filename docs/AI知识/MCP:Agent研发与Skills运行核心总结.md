LLM速通课：MCP/Agent研发与Skills运行核心总结

本次课程系统性地演示了从基础MCP服务到具备Skills能力的智能体演进过程，重点解决了多用户隔离、上下文记忆、工具调用与能力按需加载等关键问题。

1. 基础服务构建与调试
   - 使用Express搭建了监听3000端口的/chat接口，通过body-parser解析POST请求体，实现基础对话功能。
   - 前端fetch调用时因缺少Content-Type: application/json请求头导致解析失败，属于基础工程问题。

2. 会话隔离与长上下文挑战
   - 所有用户对话共享同一history数组，存在会话混杂风险，需为每个会话建立独立沙箱实现隔离。
   - 单轮对话超100次时，因大模型token限制，初始对话内容易丢失，需结合持久化存储与向量检索技术实现长期记忆。

3. Tools封装与精准参数设计
   - Agent需从“思考”转向“行动”，通过封装Tools实现自动写入文件等系统操作。
   - 工具参数必须明确定义，如生成诗文需强制要求“标题”和“作者”字段，模型会主动追问缺失项以确保准确性。
   - 类似财务报销场景，必须要求上传附件、日期、报销人等必填参数，否则无法触发执行。

4. Skills架构：解决MCP孤岛效应
   - 多MCP服务拆分导致跨部门协作低效、通用能力复用困难、资源浪费，形成“孤岛效应”。
   - Skills作为可插拔能力模块，按需加载，类比前端组件化思想，实现知识库与Tools的动态组合。

5. 三层披露式协议实现
   - Skills采用Header、Instruction、Resource三层架构：Header始终加载用于技能发现，Instruction按需加载，Resource仅在触发时执行。
   - 主流SDK（如OpenAI）尚未原生支持Skills参数，需通过自研SDK将Skills元数据注入system prompt实现兼容。
   - 实测验证：提问“总结会议”仅加载会议相关Skills元数据，不触发脚本；提问“今天天气”则完全不加载无关技能。


   会议总结

本次会议围绕“Skills 封装与智能体（Agent）运行机制”展开，六弦系统讲解了如何通过渐进式披露和按需加载机制，实现大模型与外部技能的动态交互。核心思想是：不一次性暴露全部技能细节，而是根据用户意图动态加载相关技能描述，从而降低系统复杂度与 token 消耗。

1. Skills 封装机制：系统通过扫描本地 skills 目录，提取每个技能的元数据（如 Markdown 描述），构建初始系统提示词，告知大模型当前可调用的技能列表。当用户提问涉及特定技能时，系统再动态加载该技能的完整文档（如脚本、知识库）作为上下文补充。

2. 渐进式披露与按需加载：大模型仅在识别到用户意图与某技能关联时，才返回结构化响应（如 {&amp;quot;name&amp;quot;: &amp;quot;weather&amp;quot;}），由运行时引擎解析后加载对应技能详情，避免一次性注入大量文本，有效控制上下文长度。

3. 运行时引擎（Runtime Core）：该引擎是 Agent 的“身体”，负责调度技能调用、管理多轮对话、处理沙箱与权限控制。它接收大模型返回的技能名称，查找并执行对应函数，是连接大模型与外部工具的核心胶水层。

4. 扩展性与工程实践：Skills 可部署于远程，通过拉取 ZIP 包动态加载；支持多模块（如会议、天气、PDF 生成）独立管理；建议开发者理解底层原理后，再基于开源 Agent 框架进行工程化封装。

5. 后续课程规划：后续将深入讲解 Agent 的框架层（沙箱、权限、持久化、性能优化）与真实企业落地场景（如微前端集成、简历优化），强调理解底层原理对职业转型的重要性。
